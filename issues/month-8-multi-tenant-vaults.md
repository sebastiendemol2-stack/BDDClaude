# Month-8 Multi-tenant Vaults - Detailed Tasks

M8 starts the multi-tenant vault workstream with a non-destructive bootstrap. The first slice creates tenant metadata and membership tables only. Existing vault tables keep their current shape until the tenant isolation policy is reviewed.

## Tasks

| ID | Description | Owner | Status |
|----|-------------|-------|--------|
| 1 | **Tenant bootstrap schema** - create `vault_tenants` and `vault_tenant_members` with a default `personal` tenant | Backend | DONE |
| 2 | **Static security tests** - verify RLS uses `TO` clauses, no `auth.role()`, no `user_metadata`, explicit grants | QA | DONE |
| 3 | **Documentation** - create `wiki/Intelligence/multi-tenant-vault.md` with rollout and RLS plan | Tech Writer | DONE |
| 4 | **Tenant columns plan** - decide exact tables and default tenant backfill order before altering production tables | Backend/Security | DONE |
| 5 | **Tenant id backfill migration** - add nullable `tenant_id`, index scoped tables, backfill existing rows to `personal` without RLS flip | Backend | DONE |
| 6 | **RLS migration** - enforce tenant membership checks on vault data tables after app reads are tenant-aware | Security | DONE |
| 7 | **Provisioning API** - create Edge Functions for tenant create and member assignment | Backend | DONE |
| 8 | **Export pipeline** - export tenant-scoped vault rows and markdown bundle | DevOps | DONE |
| 9 | **Import pipeline** - validate bundle, resolve conflicts, import into target tenant | DevOps | DONE |
| 10 | **Dashboard tenant selector** - tenant-aware filters and UI state | Frontend | DONE |
| 11 | **End-to-end validation** - create tenant A/B, export/import, verify isolation and link integrity | QA | DONE |

## Acceptance Criteria

- The first M8 migration is additive and safe to apply before tenant enforcement.
- Public Data API access is explicit through `GRANT`, not implied by Supabase defaults.
- RLS policies avoid deprecated `auth.role()` and never authorize from user-editable metadata.
- A default `personal` tenant exists for current single-user data migration.
- No existing vault table receives `tenant_id` until the migration/backfill plan is reviewed.

## Notes

- Supabase May 2026 changes make explicit grants important for newly created public tables.
- Tenant authorization should use `auth.uid()` and membership rows, not JWT `user_metadata`.
- The future migration should backfill `tenant_id` before changing policies on high-traffic tables.
- Tenant isolation/backfill plan is tracked in `issues/month-8-tenant-isolation-plan.md`.
- Tenant id migration is generated by Supabase CLI as `supabase/migrations/20260524095821_tenant_id_backfill_personal.sql`; it is additive only and has not been applied in this session.
- Provisioning Edge Functions (`tenant-create`, `tenant-member-assign`) share `supabase/functions/_shared/tenants.ts`. `tenant-create` is platform-admin only and seeds an `owner` member when `owner_user_id` is provided (compensating delete on member-insert failure). `tenant-member-assign` accepts platform admins or active `owner`/`admin` members of the tenant; assigning the `owner` role requires platform admin or the original `vault_tenants.owner_user_id`. Authorization never reads `user_metadata` / `raw_user_meta_data` / `auth.role()`. Functions are not deployed yet.
- Phase 5 reads: `query_vector_hybrid` accepts an optional `tenant_id` in `req` (back-compat preserved when absent). `rag-answer` resolves `body.tenant_slug` (default `personal`) against active `vault_tenants` rows and forwards `tenant_id` to the RPC. `_scripts/sync.py` resolves the active tenant lazily, sends `tenant_id=eq.<id>` on REST reads, stamps every pushed row with `tenant_id`, and refuses notes with `sensitivity ∈ {private, sensitive}`. `_scripts/rag_dashboard.py` accepts `--tenant <slug>` (default env `VAULT_TENANT` or `personal`, `all` to aggregate).
- Bundle pipeline: `_scripts/tenant_bundle.py` ships `export`/`import` subcommands. Bundle v1 = `manifest.json` (schema 1.0.0) + `tables/{vault_sections,vault_entries}.jsonl` + `markdown/<obsidian_path>` round-trippable. Import rewrites `tenant_id`, strips `id`/`created_at`/`updated_at`, blocks cross-tenant `obsidian_path` collisions unless `--on-conflict=overwrite`. Embeddings/chunks/snapshots deferred to v2.
- Acceptance gate: `_scripts/tenant_duplicate_check.py [--json] [--strict]` reports NULL `tenant_id` and composite duplicates on `vault_sections(slug)`, `vault_entries(obsidian_path)`, `projects(slug)`. Exit 0 = ready to flip uniqueness.
- E2E: `_scripts/tenant_e2e_validation.py [--keep] [--source-slug <s>] [--target-slug <s>]` provisions two tenants, seeds entries, runs export→import, asserts isolation via `query_vector_hybrid`, then tears everything down. Refuses to run against `personal`.
- RLS flip: `supabase/migrations/20260525130000_tenant_rls_enforcement.sql` (transactional). Sets `tenant_id NOT NULL`, replaces global UNIQUEs with `(tenant_id, …)` on `vault_sections.slug`, `vault_entries.obsidian_path`, `projects.slug`, adds membership-based SELECT + FOR ALL policies for `authenticated`, keeps `service_role` FOR ALL, **revokes anon** on every flipped table. Covers 15 scoped tables. Global registries (`vault_models`, `tools_registry`, `resources_registry`, `context_profiles`, `agent_permissions`) intentionally untouched. Not applied yet — apply only after the acceptance-gate script passes.
